## 推荐分工

不建议简单分成“一个人只做前端、一个人只做数据库、一个人做其余全部”。Gradio 前端本身工作量相对较小，而 GraphRAG、文档处理和模型调用工作量更大。更合理的方式是：

| 成员   | 核心角色                | 工作量 |
| ---- | ------------------- | --: |
| 成员 1 | Gradio 前端、系统集成、测试部署 | 30% |
| 成员 2 | GraphDB、知识图谱、图检索    | 35% |
| 成员 3 | 文档处理、RAG、Agent 与大模型 | 35% |

项目要求不仅考查业务实现，也单独考查可视化、数据库设计、团队进度和工程文档，因此不能把测试、文档和集成完全留到最后。

   * 调用知识库查询接口

   * 准备测试问题

## 成员 2：GraphDB 与知识图谱负责人

下文按 **Ontotext GraphDB** 设计。它使用 RDF 三元组表示数据，主要通过 SPARQL 查询，不应继续照搬原项目中的 MySQL 表和 SQLAlchemy ORM。([graphdb.ontotext.com][2])

### 主要任务

1. **知识图谱模式设计**

建议至少设计以下节点类型：

```text
Document        文档
Chunk           文档块
Entity          实体
Concept         概念或主题
Question        用户问题
Answer          系统回答
```

建议关系：

```text
Document contains Chunk
Chunk mentions Entity
Entity relatedTo Entity
Entity belongsTo Concept
Answer supportedBy Chunk
Answer usesEntity Entity
Chunk nextTo Chunk
```

实际保存为 RDF 三元组，例如：

```text
<doc_001>  rag:contains       <chunk_001>
<chunk_001> rag:mentions      <entity_人工智能>
<entity_人工智能> rag:relatedTo <entity_机器学习>
```

2. **GraphDB 环境与仓库**

   * 安装或通过 Docker 启动 GraphDB
   * 创建 Repository
   * 设置命名空间
   * 编写初始化脚本
   * 检查 GraphDB 服务状态

3. **GraphDB 数据操作**

   * 文档、分块、实体和关系写入
   * 文档删除与关联数据清理
   * 文档列表查询
   * 实体搜索
   * 根据实体查询一跳或两跳邻居
   * 查询实体之间的关系路径
   * 返回支持答案的原始文档块

4. **图检索接口**

   * SPARQL `SELECT`：查询实体、文档和关系
   * SPARQL `CONSTRUCT`：返回子图
   * SPARQL `INSERT`：写入三元组
   * SPARQL `DELETE`：删除文档相关数据
   * 编写图路径检索和结果格式化

GraphDB 还支持路径检索、全文检索和语义相似度检索。不过为了控制项目风险，第一版应优先完成“SPARQL 实体关系查询＋文档块检索”，再增加相似度索引。([graphdb.ontotext.com][3])

### 建议负责的文件

```text
graph/
├── graphdb_client.py
├── ontology.py
├── rdf_mapper.py
├── graph_repository.py
├── sparql_queries.py
└── namespaces.py

scripts/
├── init_graphdb.py

└── test_graphdb.py
```

### 对外提供的接口

```python
def save_document_graph(graph_payload: dict) -> dict:
    """保存文档、分块、实体和关系。"""

def list_documents() -> list[dict]:
    """查询知识库中的文档。"""

    """删除文档及其相关三元组。"""

    """搜索相关实体。"""

```

### 明确边界

成员 2 负责“如何建图、存图、查图”，不负责决定大模型应该如何抽取实体。实体和关系的初始抽取结果由成员 3 提供。

## 成员 3：文档处理、RAG 与 Agent 负责人

### 主要任务

1. **文档解析**

   * 读取 PDF、DOCX、TXT
   * 清洗空行和无效字符
   * 保留页码、标题、文件名等元数据
   * 文档切块
   * 生成稳定的 `document_id` 和 `chunk_id`

2. **实体与关系抽取**

   * 调用大模型识别实体
   * 抽取实体关系
   * 合并同义实体
   * 规范化关系名称
   * 输出成员 2 可以直接存入 GraphDB 的 JSON

建议输出结构：

```json
{
  "document": {
    "id": "doc_001",
    "filename": "example.pdf"
  },
  "chunks": [
    {
      "id": "chunk_001",
      "content": "……",
      "page": 3
    }
  ],
  "entities": [
    {
      "id": "entity_001",
      "name": "人工智能",
      "type": "Concept"
    }
  ],
  "relations": [
    {
      "source": "entity_001",
      "relation": "relatedTo",
      "target": "entity_002",
      "chunk_id": "chunk_001"
    }
  ]
}
```

3. **RAG 检索流程**

   * 用户问题改写
   * 意图识别
   * 提取问题中的实体
   * 调用成员 2 的图检索接口
   * 获取相关文档块和实体关系
   * 对候选内容重排
   * 组装提示词
   * 调用大模型生成答案
   * 输出引用来源

4. **Agent 路由**

至少设计三类意图：

```text
normal_chat      普通对话
document_search  文档知识问答
graph_query      实体关系或路径查询
```

推荐工作流：

```text
用户问题
   ↓
意图识别
   ├─ 普通对话 → 直接调用 LLM
   ├─ 文档问题 → 文档块检索 → 重排 → LLM
   └─ 关系问题 → 实体提取 → GraphDB 查询 → LLM
```

项目文件把 Modular RAG、GraphRAG 和 Agentic RAG 都列为可选方向，并明确提出 Query Rewriting、Knowledge Graph、Tools-Chain 和 Intent，因此这一部分应作为模型端的核心展示，而不是只完成一次简单向量查询。

## 三个人之间的接口关系

```text
                         ┌────────────────────┐
                         │ 成员1：Gradio 前端 │
                         │ 页面、状态、集成   │
                         └─────────┬──────────┘
                                   │
                         application_service
                                   │
               ┌───────────────────┴───────────────────┐
               │                                       │
     ┌─────────▼──────────┐                 ┌──────────▼─────────┐
     │ 成员3：RAG / Agent │                 │ 成员2：GraphDB     │
     │ 解析、抽取、检索   │ ──────────────► │ 存储、SPARQL、子图 │
     │ 重排、答案生成     │ ◄────────────── │ 路径、来源查询     │
     └────────────────────┘                 └────────────────────┘
```

原参考项目本身也把前端入口、文档处理、数据库、向量检索和 RAG 系统拆成独立对象，再从应用入口统一调用，因此沿用“模块解耦＋统一接口”的思路比三个人共同修改一个大文件更稳妥。

## Git 分支分工

```text
main
├── feat/gradio-ui       成员1
├── feat/graphdb         成员2
└── feat/rag-agent       成员3
```

提交格式统一：

```text
feat(ui): add document upload panel
feat(graph): implement RDF document schema
feat(rag): add intent routing
fix(graph): fix duplicate entity insertion
test(rag): add graph query test cases
docs: update deployment instructions
```

不要三个人都直接向 `main` 提交。每个功能通过合并请求进入 `develop`，联调通过后再合并到 `main`。
